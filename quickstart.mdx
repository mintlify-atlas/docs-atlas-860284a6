The Core Agent is the host-level component of VES. It tails log files, reads from systemd journals, and accepts data over sockets — then forwards enriched events to the HEIMDELL Server. This guide walks you through building from source and running your first collection pipeline.

<Note>
  Docker deployment is also available if you prefer not to build from source. See [Docker deployment](/deployment/docker).
</Note>

## Prerequisites

Before you start, make sure the following are installed on your Linux host:

| Tool | Version | Install |
|------|---------|---------|
| **Rust** | 1.89 or later | [rust-lang.org/tools/install](https://rust-lang.org/tools/install/) |
| **Git** | Any recent version | [git-scm.com/install](https://git-scm.com/install/) |
| **Protobuf compiler** (`protoc`) | Any recent version | [protobuf.dev/installation](https://protobuf.dev/installation/) |

<Tip>
  `systemd` / journald is optional. The filesystem and socket source drivers work on any Linux host. The journald source driver requires a host running systemd.
</Tip>

## Installation

<Steps>
  <Step title="Clone the repository">
    Clone the VES monorepo from GitHub:

    ```bash
    git clone https://github.com/H3IMD3LL-Labs-Inc/VES.git
    cd VES
    ```
  </Step>

  <Step title="Build the Core Agent">
    Build the `core-agent` binary in release mode. The workspace uses Rust 1.89 and the release profile enables LTO and full optimisations:

    ```bash
    cargo build --release -p core_agent
    ```

    The compiled binary lands at `target/release/core_agent`. Build time is typically 1–3 minutes on a modern machine with a warm cache.

    <Info>
      To build all workspace members (`core_agent`, `heimdell`, and `ves_lib`) in one pass, run `cargo build --release` without the `-p` flag.
    </Info>
  </Step>

  <Step title="Create a configuration file">
    The Core Agent reads its configuration from a TOML file. Create a minimal config that:

    - Identifies the agent on your network
    - Enables a filesystem source watching `/var/log`
    - Configures LMDB-backed storage for crash recovery

    ```toml config.toml
    # ── Agent identity ──────────────────────────────────────────────────────────
    [agent.identity]
    node_id   = "my-host-agent-01"
    hostname  = "my-host"
    environment = "production"

    # ── Runtime tuning (optional — shown with sensible defaults) ─────────────
    [agent.runtime]
    processor_worker_threads = 4
    max_parallel_sources     = 8

    # ── Data batching ─────────────────────────────────────────────────────────
    [agent.data_batching]
    max_events      = 1000
    max_batch_bytes = 1048576   # 1 MiB
    flush_interval  = { secs = 5, nanos = 0 }

    # ── Filesystem source: tail everything under /var/log ────────────────────
    [[sources.filesystem]]
    id        = "varlog"
    enabled   = true
    paths     = ["/var/log"]
    recursive = true

    # Only collect plain-text log files; skip compressed archives
    include_patterns = ["*.log", "*.txt"]
    exclude_patterns = ["*.gz", "*.zip"]
    file_types       = ["log"]

    # ── Storage: persist watcher/tailer checkpoints to disk ──────────────────
    [storage]
    data_dir       = "/var/lib/ves/core-agent"
    storage_engine = "lmdb"
    max_disk_bytes = 10737418240   # 10 GiB

    [storage.cleanup]
    interval_secs    = 3600   # run cleanup every hour
    max_period_secs  = 604800 # discard data older than 7 days
    ```

    Create the storage directory before starting the agent:

    ```bash
    sudo mkdir -p /var/lib/ves/core-agent
    ```

    <Warning>
      The `storage.data_dir` directory must exist and be writable by the user running the agent. The validator will reject the config and refuse to start if the path is missing or read-only.
    </Warning>
  </Step>

  <Step title="Run the Core Agent">
    Start the agent, pointing it at the config file you just created:

    ```bash
    ./target/release/core_agent --config config.toml
    ```

    For a quick test without installing, you can run directly through Cargo:

    ```bash
    cargo run --release -p core_agent -- --config config.toml
    ```
  </Step>

  <Step title="Verify collection is working">
    On startup the Core Agent logs its lifecycle transitions to `stdout`. You should see output similar to:

    ```text
    INFO  core_agent: lifecycle state -> Starting
    INFO  core_agent::config: loaded config from "config.toml" (provider=StaticFile)
    INFO  core_agent::config: config version abc123ef — applying
    INFO  core_agent::sources::filesystem: source driver "varlog" starting
    INFO  core_agent::sources::filesystem: discovered /var/log/syslog (inode=131072, offset=0)
    INFO  core_agent::sources::filesystem: discovered /var/log/auth.log (inode=131073, offset=0)
    INFO  core_agent: lifecycle state -> Running
    ```

    Each log file under `/var/log` that matches your `include_patterns` is discovered and a `Tailer` is spawned per inode. As new lines are appended to those files, they are read and forwarded downstream as `SourcePayload` events.

    When a file is rotated (for example, by `logrotate`), the Watcher detects the new inode and automatically starts a fresh Tailer for the rotated file — no restart required.

    <Tip>
      To produce test log lines immediately, run `logger "hello from VES"` in another terminal. This appends a line to the system journal and, depending on your `rsyslog` or `syslog-ng` config, to `/var/log/syslog` as well.
    </Tip>
  </Step>
</Steps>
